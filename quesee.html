QUESEE - A DIFFERENT KIND OF TEST AUTOMATION TOOL

PROBLEM TO SOLVE

In the first days from starting on a new job, I got the task to fix some of the issues the company had with an internal test automation tool. The problems were not trivial and were mostly created by the large amounts of data loaded in the web page, the persistence mechanism that used a lot of space on the drive and the comparison algorithms that did not cover all the cases and sometimes gave false positives.

There was no other way around this but to rewrite a significant part of the application to optimize the traffic between the server and the browser and also to persist smaller and more flexible data structures.

I thought this was the perfect opportunity to propose a new design of the entire application, including a brand new user interface. I can say I was lucky to get the approval for such an ambitious plan, given the fact that most companies don't invest more than necessary in their internal tools and on top of that, I had zero web development experience at that time.

In a nutshell, the application is a record/replay tool for the communication between the components of the system. The communication consists of serialized Java objects that are passed between components on specific requests. The tool is based on the idea that recordings will be done when the system is in a stable state and compared against the traffic within the system at a future date to see if there are changes.

For those of you not familiar with test automation, I will describe it to you in a nutshell.

The needs of such a testing tool gravitate along these lines :

  • Record communication
  • Replay and compare baseline vs actual results
  • Visualize comparison results, there are three types of results : success, failure and ignored)
  • Investigate the failures
  • Fix tests that are providing a wrong assessment

There are some notions that need to be defined before going further :

API description of an interaction with the service, defined by the name of the request, the type of input it needs and the type of the output it provides
API call a specific request issued to the system through an API.
Component a collection of APIs grouped to service requests of a similar nature.
Scenario a collection of API calls that are needed to support a scenario that the user goes through.
Suite collection of scenarios grouped around a similar need or functionality

SKETCHES

So I started by sketching some ideas I had after discussing with the original author of the application where he explained to me the purpose of the tool and some of the ways it is being used by the Quality Assurance team.

I played around the idea of having a tree of items on the left and a larger area on the right to display more information about the selected item from the tree.

The tree would display two different groupings of the collection of tests (by Component and by Suite) and would also provide contextual actions on mouse over for each type of item.

I also wanted to have the most important actions like recording, replaying and reports easy to access and make the state of the application visible all the time so the users will get better feedback and feel in control of the situation.

These drawing may not make too much sense for now but things will become more clear in the next section so please bear with me.

WIREFRAMES

After having a clearer idea of how I wanted the application to look like, I created a series of wireframes in order to communicate with the remote team I was working for and ask for their feedback.

One of the most important design decision that I took to solve the performance issue was to represent the objects in a hierarchical form, as a tree, and use a progressive disclosure approach when loading the data in the web page. This way, even though the user deals with large objects, expanding a node from the tree will only load the closest descendants, minimizing the traffic and making the page snappier. This is also an improvement from a usability stand point since the user has now a better overview of the data and can choose more easily what part of the object to inspect.

As you can see, the wireframes are not complete and do not cover all the possible scenarios, but this was enough to get the conversation started, get some feedback and start to develop the new user interface.

I used Balsamiq for creating the wireframes and it worked great for this project.

IMPLEMENTATION

Starting from the wireframes and from the feedback I got from the users, I began to implement the design. There was no hi fidelity prototype stage since there was no need for a more polished version and more that that, I liked the look and feel of the wireframes that make for a flat UI without bells and whistles but also with a scent of playfulness and friendliness coming from the round corners and lively colours.

It took more than a few months for the design to come to life and there were a lot of tiny additional design decisions that were made along the way to accommodate all the requirements.

Visualizing and Editing tests

Some of the screens are still not implemented but I added them anyway to showcase a flow and to see how the user drills down the test explorer section from the left.

Please notice the contextual actions that come to life when the mouse is over a row and how they differ from situation to situation (e.g. the test explorer has actions for running, ignoring and deleting tests while the object explorer has actions for shuffling collections and reverting changes).

Modifying a test is necessary in situations where the expected result has changed over time and needs to be updated. To do that, the user has to double click an existing value and type in a new one instead, technique also known as in-place editing. Please notice the consistency of the visual tags and the save button that appears only when needed.

Test Execution History

The colored chart displays the history of the test executions. Besides displaying information, it is also used for selecting a particular test execution in order to investigate the reasons behind the result. By selecting a test execution, a comparison tree is displayed right bellow the chart.

I have written a different article about the history chart since there is an interesting design story behind it.

Recording and Replaying tests

The whole idea behind the two most important features of the application was to provide immediate feedback and have the status of the application visible all the time. I used animation to convey the busy state of the system by the way of spinning icons. This is less noticeable in the screenshots than it is in the real application.

Comparing Objects

The comparison view leverages the object explorer view and decorates the tree elements with visual annotations in order to convey the changes.

In order to make the comparison work along the progressive disclosure nature of the tree, the changes need to bubble up so that the user knows how to drill down in order to find them. This is done by adding a red dotted underline to the element that needs to be expanded in order to find the change. I used the same visual cue found in most spell checkers to convey that something is wrong.

This is very different from the side by side view of the previous version of the application and it has both advantages and disadvantages. The pros are the ease of implementation and integration in the new design. The con is the color coding that needs to be learned by the users.

CONCLUSION

The tool has been adopted by the Quality Assurance team and it has been in use for most than a year at the time I wrote this article.

Most of the complaints were related to the server functionality. There were also some issues with the user interface, like the history chart that became unusable until redesigned and the colours used in the comparison view that were creating confusion for some of the users.

Even though the tool is in a pretty good shape, there are some improvements that are requested by the users and will probably be implemented in the future:

  • a side by side comparison view
  • a way of editing the expected result directly from the comparison view
  • hide the actions that don't make sense in some scenarios
  • drag and drop support in the test explorer tree for moving scenarios between suites
  • search and filter the test explorer tree
  • display reports as a hierarchy instead of a flat list

All in all, I had a great time designing, implementing and improving this tool and I sure learned a lot during the process.
