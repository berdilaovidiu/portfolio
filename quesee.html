QUESEE

PROBLEM TO SOLVE

In the first days from starting with a new job, I got the task to fix some of the bugs they had with an internal test automation tool. The problems were not trivial and were mostly created by the large amounts of data that were loaded in the web page, the persistence mechanism that used a lot of space on the drive and the comparison algorithms that did not cover all the cases and gave false positives.

There was no other way around this but to rewrite a significant part of the application to optimize the traffic between the server and the browser and also to persist smaller and more flexible data structures.

I thought this was the perfect opportunity to propose a new design of the entire application, including a brand new user interface. I can say I was lucky to get the approval for such an ambitious plan, given the fact that most companies don't invest more than necessary in their internal tools and on top of that, I had zero web development skills at that time.

In a nutshell, this is a record/replay tool for the communication between the components of the system. The communication consists of serialized Java objects that are passed between components on specific requests. The tool is based on the idea that recordings will be done when the system is in a stable state and those recordings will be compared against the traffic within the system at a future date to make sure nothing changed or to spot what changed.

For those of you not familiar with test automation, I will describe it to you in a nutshell.

The needs of such a testing tool gravitate along these lines :

  • Record
  • Replay and compare
  • Visualize results
  • Investigate failures
  • Fix tests that are providing wrong assessment

There are some notions that need to be defined before going further :

  • API - description of an interaction with the service defined by the name of the request, the kind of the input it needs and the type of the output it provides
  • API call - a specific request issued to the system through an API.
  • Component - a collection of APIs grouped to service requests of a similar nature.
  • Scenario - a collection of API calls that are needed to support a scenario that the user goes through.
  • Suite - a collection of scenarios grouped around a similar need or functionality

SKETCHES

So I started by sketching some ideas I had after having a discussion with the original author of the application where he explained to me the purpose of the tool and some of the ways it is being used by the Quality Assurance team.

I played along the idea of having a tree of items on the left and a larger area on the right to display more information about the selected item from the tree.

The tree would display two different groupings of the collection of tests (by Component and by Suite) and would also provide contextual actions on mouse over for each type of item.

I also wanted to have the most important actions like running, recording and reports easy to access and have the state of the application visible at all times.

WIREFRAMES

After having a clearer idea of how I wanted the application to look like, I created a series of wireframes in order to communicate with the remote team I was working for and ask for their feedback.

As you can see, the wireframes are not complete and do not cover all the possible scenarios, but this was enough to get the conversation started, get some feedback and start to develop the new user interface.

IMPLEMENTATION

Starting from the wireframes and from the feedback of the users, I began to implement the design. There was no hi fidelity prototype since there was no need of a polished version and more that that, I liked the look and feel of the wireframes that suggested a flat UI with no bells and whistles but also with a feel of playfulness and friendliness coming from the round corners and alive colours.

It took more than a few months for the design to come to life and there were a lot of tiny additional decisions that were made along the line to accommodate all the requirements.

CONCLUSION

The tool is now being used by the QA team

Most of the complaints were about the back-end functionality. There were some things related to the UI, like the history chart that became unusable until redesigned and the colours that were being used in the comparison view
